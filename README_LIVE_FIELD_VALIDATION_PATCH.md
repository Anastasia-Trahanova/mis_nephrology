# Live field validation patch

## Назначение

Патч переводит предупреждения формы из сценария «нажали сохранить → белая страница с JSON» в сценарий «врач ввёл неверное значение → поле сразу подсветилось».

Текущий дизайн страниц не меняется. Изменения видны только при ошибке поля.

## Файлы

### `app/static/js/simple_form_guard.js`

Главный файл живой проверки формы.

Проверяет:

- обязательные поля;
- буквы в числовых клинических полях;
- отрицательные / абсурдно маленькие / абсурдно большие значения;
- связь систолического и диастолического АД;
- дату рождения в будущем;
- дату приёма раньше даты рождения;
- дату следующего визита раньше даты приёма;
- дату следующего визита раньше текущей даты;
- дату исследования позже даты приёма.

Показывает только короткие сообщения:

- `Неверное значение`
- `Не все обязательные поля заполнены`
- согласованные короткие сообщения по датам

Не показывает врачу технические пределы проверки и не показывает сообщение про нормализацию `1015 → 1.015`.

### `app/static/css/02_clinical_messages.css`

Файл внешнего вида сообщений.

Здесь можно менять:

- цвет красной рамки поля;
- цвет текста ошибки под полем;
- размер текста ошибки;
- фон общего сообщения над формой;
- оформление нейтральных подсказок под расчётными таблицами.

### `app/templates/_clinical_validation.html`

Серверный fallback для будущего возврата ошибок внутрь формы.

Основная live-валидация идёт через JS, но partial оставлен, чтобы позже не показывать врачу технический JSON, если сервер всё-таки вернул ошибки.

### `scripts/apply_live_validation_patch.py`

Точечный скрипт подключения.

Делает:

1. подключает `simple_form_guard.js` в `base.html`, если он ещё не подключён;
2. подключает `02_clinical_messages.css` в `base.html`, если он ещё не подключён;
3. убирает серверную блокировку частично заполненной ACR-пары;
4. чуть упрощает fallback-сообщения в `validation.py`.

Скрипт можно запускать повторно.

### `tests/layer/test_live_field_validation_assets.py`

Проверяет, что frontend-валидация содержит нужные короткие сообщения, правила и классы.

### `tests/browser/test_live_field_validation.py`

Опциональный browser-тест. Проверяет, что при вводе `abc` в поле систолического АД появляется `Неверное значение` рядом с полем.

## Как применить

1. Распаковать архив в корень проекта с заменой файлов.
2. Запустить:

```cmd
python scripts\apply_live_validation_patch.py
```

3. Перезапустить сервер:

```cmd
python -m uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

4. Обновить страницу через `Ctrl+F5`.

## Проверка

```cmd
pytest tests/layer/test_live_field_validation_assets.py
pytest tests/layer
```

Browser-проверка:

```cmd
set RUN_BROWSER_TESTS=1
set APP_BASE_URL=http://127.0.0.1:8000
set E2E_LOGIN=admin
set E2E_PASSWORD=admin123
set E2E_EXISTING_PATIENT_ID=1
pytest -rs tests/browser/test_live_field_validation.py
```

## Ручная проверка

Открыть форму повторного приёма или нового пациента.

Проверить:

- ввести `abc` в систолическое АД → рядом появляется `Неверное значение`;
- ввести дату рождения завтра → появляется `Дата рождения не может быть в будущем`;
- ввести дату исследования позже даты приёма → появляется `Дата исследования не может быть позже даты приёма`;
- нажать сохранить с пустым врачом/отделением → появляется `Не все обязательные поля заполнены`, форма прокручивается к полю;
- ввести удельный вес `1015` → предупреждение не появляется;
- ввести удельный вес `1500` → появляется `Неверное значение`.

## Важное ограничение

Этот патч предотвращает основной неприятный сценарий через live-проверку до отправки формы.

Полное устранение белой страницы с JSON на уровне сервера лучше сделать отдельным этапом: POST-роуты должны при ошибке возвращать ту же HTML-форму с введёнными данными, а не `HTTPException` с JSON. Это более крупная правка, потому что надо аккуратно восстановить контекст формы, врачей, отделения и уже введённые значения.
