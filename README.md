# МИС Нефролога

МИС Нефролога — учебно-практический веб‑проект медицинской информационной системы для ведения пациентов с хронической болезнью почек. Сейчас проект развивается как MVP для регистрации пациентов, ведения приёмов, хранения клинических данных, расчётов нефрологических показателей, формирования карточки пациента и подготовки экспортов.

Проект не является зарегистрированным медицинским изделием. Алгоритмы и расчёты должны использоваться как программная поддержка и требуют врачебной интерпретации.

## Текущее состояние проекта

В текущей версии уже сделаны основные архитектурные этапы:

- FastAPI-приложение разделено на роутеры, сервисы и слой repositories.
- Большой сценарий пациента/приёма вынесен из одного файла в отдельные модули.
- SQL-запросы постепенно отделены от бизнес-логики.
- Добавлены клинические алгоритмы расчёта ИМТ, СКФ, стадии ХБП, ACR и прогноза KDIGO.
- Добавлена нормализация клинических числовых значений: запятая в десятичных числах, пробелы, безопасная нормализация удельного веса мочи `1015 -> 1.015`, гематокрит `0.39 -> 39`.
- Добавлена серверная и клиентская проверка абсурдных значений и дат.
- Форма приёма получила live-подсветку ошибок: врач видит проблему рядом с полем до отправки формы.
- Карточка пациента разделена на partial-шаблоны без изменения исходного расположения блоков и кнопок.
- Добавлен слой CSS-настроек для будущего единообразного дизайна.
- Добавлены layer-тесты, опциональные интеграционные тесты с БД и browser-тесты.

## Основные возможности

- список пациентов и поиск;
- добавление нового пациента;
- добавление повторного приёма с подгрузкой данных последнего приёма;
- карточка пациента с историей приёмов;
- осмотр, жалобы, анамнез, физикальные данные;
- ОАК, биохимия, ОАМ, альбуминурия, УЗИ;
- расчёт BMI, СКФ CKD-EPI 2021, Cockcroft–Gault, стадии ХБП;
- расчёт ACR и категории альбуминурии A1/A2/A3;
- прогноз ХБП по KDIGO;
- диагнозы МКБ-10;
- диета, рекомендации, лекарственные назначения;
- экспорт выбранного приёма в Word;
- регистр ХБП.

## Структура проекта

```text
app/                  Основной код приложения
  routers/            HTTP-роуты FastAPI
  services/           Бизнес-логика и сборка данных приёма
  repositories/       SQL-операции и работа с БД
  medical_algorithms/ Медицинские расчётные функции
  templates/          Jinja2-шаблоны страниц и partial-файлы
  static/             CSS и JavaScript

database/             SQL-файлы базы данных и инструкции по БД
migrations/           Alembic-миграции
scripts/              Скрипты запуска, тестирования и обслуживания
tests/                Автоматические тесты
docs/                 Ручные чек-листы и проектная документация
```

Инструкции по базе данных находятся в папке `database/`. Их не нужно переписывать при обычной правке документации проекта.

## Быстрый запуск приложения

Из корня проекта:

```cmd
cd /d D:\python\ckd\mis_for_registrations
D:\python\ckd\mis_fastapi\venv\Scripts\activate
python -m uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

Если используется локальный `venv` внутри проекта, можно активировать его обычным способом:

```cmd
venv\Scripts\activate
python -m uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

После запуска открыть:

```text
http://127.0.0.1:8000
```

## Настройки окружения

Локальный `.env` не хранится в Git. Он должен находиться в корне проекта и содержать реальные параметры подключения.

Шаблон хранится в `.env.example`:

```env
DB_HOST=localhost
DB_PORT=5432
DB_NAME=mis_for_registrations
DB_USER=postgres
DB_PASSWORD=your_password_here
DB_POOL_MIN_CONN=1
DB_POOL_MAX_CONN=10
SESSION_SECRET_KEY=dev-local-secret
```

Если приложение падает с ошибкой по `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_PASSWORD` или `SESSION_SECRET_KEY`, нужно проверить наличие `.env` в корне проекта.

## База данных

Основная инструкция по базе лежит в `database/README.md` и дополнительных README внутри `database/`. Эти файлы считаются рабочей инструкцией по восстановлению БД и не должны удаляться при уборке документации.

Короткий dev-сценарий пересоздания базы через скрипт:

```cmd
python scripts\reset_db.py --yes
```

Для базы без демо-данных:

```cmd
python scripts\reset_db.py --yes --no-demo
```

Для применения миграций без удаления базы:

```cmd
python scripts\reset_db.py --no-drop --no-demo
```

## Тесты

Быстрая безопасная проверка без БД и без запущенного сервера:

```cmd
pytest tests/layer
```

Проверка интеграционного сценария с dev/test БД:

```cmd
set RUN_DB_LAYER_TESTS=1
pytest tests/integration/test_patient_appointment_database_workflow.py
```

Browser-тесты при запущенном приложении:

```cmd
set RUN_BROWSER_TESTS=1
set APP_BASE_URL=http://127.0.0.1:8000
set E2E_LOGIN=admin
set E2E_PASSWORD=admin123
set E2E_EXISTING_PATIENT_ID=1
set E2E_EXISTING_APPOINTMENT_ID=1
pytest -rs tests/browser
```

Установка Playwright при первом запуске browser-тестов:

```cmd
pip install playwright
python -m playwright install chromium
```

## Клиническая валидация и live-подсветка

Файл `app/validation.py` отвечает за серверную проверку. Файл `app/static/js/simple_form_guard.js` отвечает за подсветку ошибок в браузере до отправки формы.

Врач видит короткие сообщения:

- `Неверное значение`;
- `Не все обязательные поля заполнены`;
- `Дата рождения не может быть в будущем`;
- `Дата приёма не может быть раньше даты рождения`;
- `Дата следующего визита не может быть раньше даты приёма`;
- `Дата следующего визита не может быть раньше текущей даты`;
- `Дата исследования не может быть позже даты приёма`.

Технические диапазоны и детали нормализации не выводятся врачу.

## Стадии ХБП

В проекте используется русская буква `С` в стадиях ХБП:

```text
С1, С2, С3а, С3б, С4, С5
```

Это важно не менять на латинскую `C`/`G` без осознанной миграции данных и тестов.
