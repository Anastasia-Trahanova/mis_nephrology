# README: создание Alembic-миграции и запуск базы одной командой

Этот файл описывает, что было сделано для настройки миграций БД проекта МИС ХБП/нефрологии, какие файлы за что отвечают и какие команды мы запускали.

Цель: воспроизводимое создание базы через Alembic и Python-команду.

---

## 1. Что мы хотели получить

Изначально база создавалась вручную через DBeaver и отдельные SQL-файлы.

Теперь нужно, чтобы базу можно было поднять на новом компьютере так:

```cmd
python scripts\reset_db.py --yes
```

Эта команда должна:

1. удалить тестовую базу, если она уже есть;
2. создать базу заново;
3. применить Alembic-миграции;
4. создать таблицы, связи, ограничения, справочник МКБ и view;
5. загрузить тестовые данные;
6. проверить, что база создалась корректно.

---

## 2. Текущая структура проекта

Минимальная структура проекта для работы миграций:

```text
mis_for_registrations/
  database/
    01 создание таблиц.sql
    02 настройка связей ключей и ограничений.sql
    03 создание и заполнение таблиц МКБ.sql

    05 оценка целостности.sql
    06 аналитические запросы.sql

  migrations/

  scripts/
    reset_db.py

  alembic.ini
  requirements.txt
  README_МИГРАЦИИ_И_ЗАПУСК.md
```

## 3. Зачем добавили Alembic

Alembic нужен не для того, чтобы заменить SQL-файлы, а чтобы потом можно было обновлять уже существующую базу без пересоздания.

Например:

```text
0001_baseline_schema.py       — первая структура базы
0002_add_audit_log.py         — будущая таблица аудита
0003_add_algorithm_versions.py — будущая таблица версий медицинских алгоритмов
```

Когда база уже существует у врача или на тестовом сервере, нельзя каждый раз делать `DROP DATABASE`. Нужно применять изменения аккуратно:

```cmd
alembic upgrade head
```

## 4. Какие файлы Alembic за что отвечают

### `alembic.ini`

Лежит в корне проекта. Это главный конфиг Alembic. В нем указано, что папка миграций находится здесь:

```ini
[alembic]
script_location = migrations
```

Строку подключения с паролем в `alembic.ini` не храним. Ее передает `reset_db.py` через переменную окружения `DATABASE_URL`.

---

### `migrations/env.py`

Это файл, который Alembic запускает перед применением миграций. Он получает подключение к базе из переменной окружения:

```python
env_url = os.getenv("DATABASE_URL")
```

Если `DATABASE_URL` не задан, он может собрать строку подключения из отдельных переменных:

```python
DB_HOST
DB_PORT
DB_USER
DB_PASSWORD
DB_NAME
```

Главная идея: пароль не должен быть жестко записан в коде.

---

### `migrations/script.py.mako`

Это шаблон для будущих миграций. Когда будет команда:

```cmd
alembic revision -m "add audit log"
```

Alembic создаст новый файл миграции в `migrations/versions/` на основе этого шаблона. Обычно этот файл руками не редактируется.

---

### `migrations/versions/0001_baseline_schema.py`

Это первая миграция. Она создает базовую структуру базы из наших SQL-файлов. Внутри она выполняет:

```python
op.execute(_read_sql("01 создание таблиц.sql"))
op.execute(_read_sql("02 настройка связей ключей и ограничений.sql"))
op.execute(_read_sql("03 создание и заполнение таблиц МКБ.sql"))
```

То есть baseline-миграция включает:

1. таблицы;
2. внешние ключи, ограничения, индексы;
3. таблицы МКБ;
4. заполнение справочника МКБ;
5. view `appointment_icd10_diagnoses_view`.

Важно: тестовые данные в миграцию не включены.

---

## 5. Почему тестовые данные не стали отдельной миграцией

Мы решили так:

```text
Миграции = структура базы + справочники
Тестовые данные = отдельное dev/demo-наполнение
```

Поэтому файл:

```text
04 заполнение тестовыми данными.sql
```

запускается не Alembic-миграцией, а скриптом:

```text
scripts/reset_db.py
```

после применения миграций. Это правильно, потому что в боевой базе тестовые пациенты не нужны.

---

## 6. Что делает `scripts/reset_db.py`

Файл:

```text
scripts/reset_db.py
```

это Python-обертка для удобного запуска. Она делает то, что раньше приходилось делать вручную:

1. спрашивает пароль от пользователя `postgres`;
2. удаляет базу `mis_for_registrations`;
3. создает чистую базу `mis_for_registrations`;
4. передает Alembic строку подключения через `DATABASE_URL`;
5. запускает:

```cmd
alembic upgrade head
```

6. загружает тестовые данные из:

```text
database/04 заполнение тестовыми данными.sql
```

7. проверяет наполнение таблиц;
8. проверяет view;
9. проверяет стадии ХБП;
10. запускает файл:

```text
database/05 оценка целостности.sql
```

## 7. Как запустить миграционное создание базы

Из корня проекта:

```cmd
cd /d D:\python\ckd\mis_for_registrations
venv\Scripts\activate
python scripts\reset_db.py --yes
```

Скрипт спросит пароль от PostgreSQL один раз. 
После успешного запуска в консоли должно быть примерно:


## 8. Как проверить, что Alembic действительно сработал

После запуска выполнить:

```cmd
psql -h localhost -U postgres -d mis_for_registrations -c "SELECT * FROM alembic_version;"
```

Ожидаемый результат:

```text
version_num
0001_baseline_schema
```

Это значит, что Alembic применил первую миграцию.

---

## 9. Как создать базу без тестовых данных

Для чистой базы без демо-пациентов:

```cmd
python scripts\reset_db.py --yes --no-demo
```

В этом режиме будут созданы:

- таблицы;
- связи;
- ограничения;
- индексы;
- МКБ;
- view.

Но тестовые пациенты и приемы не загрузятся.

---

## 10. Как применить миграции без удаления базы

Если база уже существует и ее не нужно пересоздавать:

```cmd
python scripts\reset_db.py --no-drop --no-demo
```

Этот режим полезен позже, когда в базе появятся реальные данные. Но на текущем этапе разработки мы обычно используем:

```cmd
python scripts\reset_db.py --yes
```

потому что работаем с тестовой базой.

## 11. Что было важно по кодировке

На Windows CMD перед ручным запуском SQL-файлов нужно было использовать:

```cmd
chcp 65001
set PGCLIENTENCODING=UTF8
```

Иначе файл МКБ мог падать с ошибкой кодировки. В Python-скрипте `reset_db.py` эта проблема решается проще, потому что Python нормально передает UTF-8-файлы в `psql`/PostgreSQL.

## 13. Главное резюме

Сейчас у нас есть два рабочих пути:

### Ручной путь

```cmd
psql ... -f ".\database\01 создание таблиц.sql"
psql ... -f ".\database\02 настройка связей ключей и ограничений.sql"
psql ... -f ".\database\03 создание и заполнение таблиц МКБ.sql"
psql ... -f ".\database\04 заполнение тестовыми данными.sql"
```

### Миграционный путь

```cmd
python scripts\reset_db.py --yes
```

Для дальнейшей разработки основной удобный способ:

```cmd
python scripts\reset_db.py --yes
```

Он создает базу полностью, загружает тестовые данные и проверяет, что все работает.
