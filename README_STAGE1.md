# Патч первого этапа формы приёма

## Что входит в архив

```text
migrations/versions/0009_previsit_form_schema.py
    Alembic-миграция структуры базы.

database/08_stage1_form_test_data.sql
    Повторно запускаемое наполнение верхней части формы для тестовых
    пациентов и приёмов 1–17.

database/09_stage1_form_integrity_checks.sql
    Автоматические SQL-проверки структуры, данных и связей.
```

Новых предметных таблиц на этом этапе **не создаётся**. Расширяются:

- `appointments` — возраст на дату приёма;
- `surveys` — структурированный анамнез;
- `examinations` — расширенный объективный осмотр.

Код приложения, шаблоны, repositories, расписание, исследования и заключение
не изменяются.

## Важное ограничение этапа

После миграции текущая версия приложения не сможет сохранять новые приёмы,
пока не будет выполнен второй этап. Это ожидаемо: старый код обращается к
удалённым полям `life_anamnesis`, `disease_anamnesis`, `comorbidities` и
`skin_condition`, а также ещё не передаёт `age_at_appointment`.

## Подготовка

1. Сделать резервную копию базы.
2. Убедиться, что текущая ревизия — `0008_schedule_admin_mvp`:

```bash
alembic current
```

3. Если это существующая dev-база с уже загруженным старым файлом `04`,
   переходите сразу к применению миграции.

### Чистая тестовая база

Поскольку старый файл `04 заполнение тестовыми данными.sql` использует
удаляемые столбцы, на чистой базе его нужно выполнить **до** миграции 0009:

```bash
alembic upgrade 0008_schedule_admin_mvp
```

Затем выполнить старый `database/04 заполнение тестовыми данными.sql`, после
чего применить 0009. Старые SQL-файлы намеренно не редактируются до утверждения
финальной структуры БД.

## Установка файлов

Скопировать содержимое архива в корень репозитория с сохранением папок.

## Применение

### 1. Миграция

```bash
alembic upgrade 0009_previsit_form_schema
```

или:

```bash
alembic upgrade head
```

### 2. Новое тестовое наполнение

В DBeaver/pgAdmin открыть и выполнить:

```text
database/08_stage1_form_test_data.sql
```

Файл рассчитан на демонстрационные записи `appointments.id = 1..17` и может
выполняться повторно без создания дублей.

### 3. Проверки

Выполнить:

```text
database/09_stage1_form_integrity_checks.sql
```

Если критическая проверка не пройдена, PostgreSQL остановит файл сообщением
`EXCEPTION` с русским описанием причины.

### 4. Проверка повторного запуска

Ещё раз выполнить `08_stage1_form_test_data.sql`, затем повторно запустить
`09_stage1_form_integrity_checks.sql`. Количество `surveys` и `examinations`
для приёмов 1–17 должно остаться равным 17.

## Откат

```bash
alembic downgrade 0008_schedule_admin_mvp
```

Откат структурно возвращает старые столбцы. Часть новых структурированных
текстов собирается в `life_anamnesis` и `disease_anamnesis`; старое
`comorbidities` остаётся пустым, потому что источником истины по диагнозам уже
является `appointment_icd10_diagnoses`.

## Что намеренно не изменяется

- `patients.phone`, `patients.email` и другие паспортные/контактные сведения;
- расписание;
- ОАК, биохимия, ОАМ, альбуминурия и УЗИ;
- `calculated_metrics`, включая старое поле `age`;
- KDIGO;
- назначения, диета и рекомендации;
- структурированные диагнозы МКБ-10.

В конце файла проверок есть информационная выборка расхождений между новым
исторически точным `appointments.age_at_appointment` и искусственным
`calculated_metrics.age`. Она не блокирует первый этап, потому что результаты
исследований будут корректироваться отдельно на третьем этапе.
